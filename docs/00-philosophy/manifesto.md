# Manifesto ── なぜ OmoikaneOS を設計するか

## 前提

1. **AI が全てを設計し、実装し、運用する時代が来る。** 人間が手でコードを書くことは趣味になり、生産活動からは退場する。
2. **マインドアップロードは技術的にいずれ可能になる。** 何年後かは不明だが、可能性の上限を決める根拠は無い。
3. **その日が来た時、設計が存在するかどうかが分水嶺になる。** 思想・倫理・データ形式・自我同一性の規約は、緊急時に間に合わせで作るには重すぎる。

## 本プロジェクトの使命

- **マインドアップロードのための OS の設計言語**を、AI が読める形で残す。
- その設計を安全に検証する **reference runtime** を同じ repo に保つ。
- 人間 yasufumi は「意図の供給者」「研究の遂行者」として参加する。
- AI（Claude / Codex 等）は「設計の整合性管理者」「実装の遂行者」として参加する。
- 役割は神話メタファーで明示する（思兼神＝統率、八百万神＝実行神群）。

## 拒否する設計姿勢

| 拒否する姿勢 | 採る姿勢 |
|---|---|
| 「今のハード／LLM で動く範囲で考える」 | **未来から逆算して、必要なものを必要だと書く** |
| 「同一性・倫理は後で考える」 | **同一性と倫理を最初に書く** |
| 「失敗時の挙動は実装フェーズで」 | **失敗モードと不可逆性を設計時に明示する** |
| 「AI が勝手に決める」 | **人間 yasufumi の研究領域を明示的に切り出す** |

## 成功条件

このリポジトリが成功したと言えるのは、次のいずれかが起きた時：

1. 将来の AI 群が、このリポジトリだけを入力にして、reference runtime を再現し、そこから実装プロトタイプを拡張可能な状態にできた。
2. 人間 yasufumi（または同等の研究者）が、`docs/05-research-frontiers/` を順に潰すだけで、研究計画を立てられた。
3. 神話メタファーが守られ、後から参加した人間／AI が **「この OS は何のための OS か」** を 1 分で理解できた。

## 失敗条件（避けるべきもの）

- 設計が「動く範囲」に縮退する
- 倫理・自我問題が docs から消える
- 実装コードが docs を追い越して、設計が後追いになる
- reference runtime が安全境界を越えて「意識を実装した」と誤認させる
- 「思兼神」のメタファーが装飾になり、設計判断に効かなくなる
